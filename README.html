raiveFlier

Upload a rave flier. Get the full story behind every artist, venue, and promoter — with citations.

v0.1.0 Python 3.12+ MIT License Alpha

What is raiveFlier?

raiveFlier is an AI-powered research tool that transforms rave and concert flier images into deep, citation-backed profiles of every entity on the flier. Upload a photo of a flier, and raiveFlier will:

  1. Extract text via multi-engine OCR (LLM Vision, EasyOCR, Tesseract)
  2. Identify artists, venues, promoters, dates, genres, and ticket prices using LLM analysis
  3. Let you confirm or edit extracted entities before research begins
  4. Deep-research every entity using music databases (Discogs, MusicBrainz), web search, and article scraping
  5. Map interconnections between entities and present the results with full citations

Who is it for? Music historians, journalists, DJs, collectors, archivists, and anyone curious about the people and places behind underground music events.

Quick Start

Prerequisites

  • Python 3.12+
  • pip (package manager)
  • At least one LLM provider: OpenAI API key, Anthropic API key, or Ollama running locally (free)
  • Optional: Tesseract OCR installed locally (brew install tesseract on macOS)

Setup

# 1. Create and activate a virtual environment
python3 -m venv .venv && source .venv/bin/activate

# 2. Install dependencies
pip install -r requirements.txt

# 3. Configure environment variables
cp .env.example .env
# Edit .env with your API keys (at minimum, one LLM provider)

# 4. Start the server
uvicorn src.main:app --reload

# 5. Open in browser
# http://localhost:8000

Zero-cost mode: raiveFlier works with zero paid APIs. Use Ollama for LLM (free, local), DuckDuckGo for search (free, no key), and MusicBrainz for music data (free, no key). Only Discogs API, Serper, and OpenAI/Anthropic require API keys.

Architecture Overview

raiveFlier uses a layered architecture with adapter pattern. All external services are abstracted behind interfaces, making every provider swappable without changing business logic. The analysis runs through a 5-phase pipeline with a human-in-the-loop confirmation gate between Phase 1 and Phase 2.

API Reference

All API endpoints are prefixed with /api/v1. Interactive docs available at /docs (Swagger UI) when the server is running.

Method Endpoint Description
POST /api/v1/fliers/upload Upload a flier image (JPEG/PNG/WebP, max 10MB). Returns extracted entities for review.
POST /api/v1/fliers/{session_id}/confirm Confirm (or edit) extracted entities. Kicks off background research pipeline.
GET /api/v1/fliers/{session_id}/status Poll current pipeline phase, progress percentage, and errors.
GET /api/v1/fliers/{session_id}/results Retrieve full analysis results (research profiles, interconnection map, citations).
GET /api/v1/health Health check — returns app version and provider availability.
GET /api/v1/providers List all configured providers with type and availability status.
GET /api/v1/corpus/stats RAG corpus statistics (requires RAG_ENABLED=true).
WS /ws/progress/{session_id} WebSocket for real-time pipeline progress updates.

CLI: RAG Ingestion

Ingest articles, books, and analysis documents into the RAG corpus for enriched research results.

# Ingest a single article by URL
python -m src.cli.ingest --type article --url "https://example.com/article"

# Ingest a local text file
python -m src.cli.ingest --type book --file path/to/book.txt

RAG is optional. Set RAG_ENABLED=true in .env and ensure an embedding provider (OpenAI or Ollama with Nomic) is available.
