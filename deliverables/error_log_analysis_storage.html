Critical Security Vulnerabilities

#1 XSS Vulnerability in connections.js

Critical Resolved
Location: tools/raive_feeder/frontend/js/connections.js, lines 225–237
Symptom: API-sourced strings (entity names, relationship types, session IDs) were injected directly into innerHTML without escaping.
Root Cause: _showEntityDetail() used template literals with raw API data inside innerHTML assignments. No sanitization layer existed between the API response and DOM insertion.
LLM-generated content or OCR results containing HTML/script tags could execute arbitrary JavaScript in the browser. Since raiveFlier processes user-uploaded flier images through OCR and LLM entity extraction, an attacker could craft a flier image containing malicious text that would be extracted as an entity name and rendered unsafely in the connection map UI.
Added _escapeHtml() helper that creates a temporary DOM element and uses the textContent/innerHTML roundtrip for safe escaping. Applied to all interpolated values in innerHTML calls throughout the connections module.

#2 Missing Pagination Bounds Validation

Critical Resolved
Location: src/api/routes.py, list_analyses() endpoint
Symptom: limit and offset query params accepted any integer value including negative numbers and extremely large values.
Root Cause: Parameters were declared as plain int with defaults but no FastAPI Query() constraints. No input validation was applied at the API boundary.
limit=-1 returns all rows in SQLite; very large limits could cause memory exhaustion on the 512MB Render deployment. This is a denial-of-service vector exploitable by any authenticated API consumer.
Changed parameters to use FastAPI's Query() with explicit bounds: Query(default=50, ge=1, le=200) for limit and Query(default=0, ge=0) for offset. FastAPI now rejects out-of-range values with a 422 response before the handler executes.

Logic & Dead-Code Bugs

#3 Dead-Code 404 Check in Connection Map Node Endpoint

Medium Resolved
Location: tools/raive_feeder/api/routes.py, get_connection_map_node()
Symptom: The 404 check if detail is None never triggered, meaning requests for non-existent nodes returned a 200 with a {"found": False} body instead of a proper 404 status code.
Root Cause: Mismatch between the service's return contract and the endpoint's guard check. get_node_detail() returns {"found": False} (a truthy dict), never None.
Changed the guard to if not detail or not detail.get("found"), which correctly catches both None returns and the {"found": False} dict. The endpoint now returns HTTP 404 for missing nodes as intended.
