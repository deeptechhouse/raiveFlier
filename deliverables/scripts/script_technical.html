[0:42–0:52] Slide 6 — Architecture (7-Layer) KEY POINT

0:42

“Seven-layer architecture with a strict adapter pattern. Frontend talks REST and WebSocket to the API layer. API hands off to the pipeline orchestrator. Pipeline calls services. Services talk to abstract interfaces — never to concrete providers. Providers implement those interfaces. And at the bottom, thirty-nine Pydantic frozen models. All wiring happens in a single composition root — main.py — via dependency injection. Swap OpenAI for a local Ollama instance by changing one environment variable. No vendor lock-in. Period.”

[Screen: 7-layer architecture diagram with arrows flowing top to bottom, DI callout at main.py]

KEY POINT — Architecture: adapter pattern, DI composition root, no vendor lock-in, immutable models.

[1:01–1:11] Slide 8 — RAG Technical Deep-Dive KEY POINT

1:01

“Eight-step RAG pipeline. Documents chunked into five-hundred-twelve-token sliding windows with sixty-four-token overlap, preserving paragraph boundaries. Embedded via FastEmbed — fifty megs of ONNX runtime, not the five hundred megs of PyTorch we can’t afford on a five-twelve-meg container. Stored in ChromaDB. At query time: vectorize, retrieve with cosine similarity, filter by entity metadata, over-fetch four-x, deduplicate per source — max three chunks per source — and augment the LLM prompt. Every fact gets a citation tier. T1: published academic and official sources. T2: established music press. T3: web and user-generated. The model synthesizes with inline citations. No hallucinations invited.”

[Screen: 8-step RAG pipeline diagram — ingest, chunk, embed, store, query, retrieve, augment, synthesize]

KEY POINT — RAG pipeline: chunking params, embedding choice, retrieval strategy, citation tiers (T1/T2/T3).

Production Notes

Pacing

Total runtime: 155 seconds across 18 slides. Average ~8.6 seconds per slide. The architecture slide (6), RAG deep-dive (8), and code examples (10) are the densest — allow a half-beat of silence after each to let the diagrams register. Slides 1, 4, and 18 are the emotional bookends; deliver them with slightly more emphasis.

Humor Placement

Three humor beats: “fonts designed during an earthquake” (Slide 3), “untested code is just an opinion” (Slide 4), and the two a.m. commit history line (Slide 16). Space them out — they work as tension release after technical density. The OCR/serif fonts joke (Slide 2) is the first laugh; land it cleanly.

Key Transitions

Slide 6 → 7 is the critical tonal shift: from architecture rigor to accessible analogy. The “quick non-technical framing” opener on Slide 7 signals the audience to relax briefly before Slide 8 goes deep again. Slide 14 → 15 (solved → ahead) should feel like a pivot from confidence to ambition.

Metrics to Emphasize

Let the animated counters on Slide 4 run for 2–3 seconds before narrating. The visual impact of numbers ticking up reinforces the scale claim. On Slide 16, the “nine days” reveal should land with a slight pause before it — the timeline is impressive and deserves a beat.

Fallback if Running Long

Slides 11 (UML) and 13 (RAG Improvements) can be trimmed to 5–6 seconds each if pacing runs hot. Cut adjectives, not data points.
